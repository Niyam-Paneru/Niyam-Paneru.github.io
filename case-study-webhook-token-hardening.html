DentSignal engineering case study · FastAPI / webhooks

Retries needed identity. Callbacks needed their own clock.

Missing provider event IDs weakened deduplication, while one short authentication lifetime could not safely serve both immediate admission and delayed callbacks.

Area
FastAPI, HMAC, webhooks, idempotency
Evidence
PR-reported historical evidence
Boundary
No keys, payloads, or compliance claim

01 / Problem

One trust window served two different risks.

Provider events without IDs could evade ordinary deduplication. Delayed callbacks outlived the short WebSocket-admission window, but simply extending that window would widen replay exposure.

Root cause: event identity and token purpose were not modeled precisely enough.

02 / Repair

Bind identity and lifetime to purpose.

  • Added a stable fallback identity for events without provider IDs.
  • Preserved explicit callback state transitions and failed call-level states.
  • Introduced route- and resource-bound callback tokens with a longer callback window.
  • Kept WebSocket admission short-lived and limited legacy compatibility to its original window.

03 / Validation

Regression coverage followed the trust boundaries.

PR 122 reports the full backend test directory, TypeScript, ESLint, build, and secret audit. PR 210 reports 151 focused tests, Ruff, secret scan, and an external-service guard.

Before: weaker retry identity and a lifetime mismatch. After: explicit deduplication and purpose-bound token windows at the cited revisions.

04 / Rollback

Revert the bounded backend slices.

Neither repair requires a database migration. A prior backend image or commit revert restores the earlier behavior; the compatibility window must be considered before token code is rolled back.
