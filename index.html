Process

From symptom to tested handoff.

  1. 01 · Fit check You send the symptom, expected behavior, environment, deadline and acceptance criteria—without secrets.
  2. 02 · Reproduce I confirm the failure and write a narrow acceptance condition before changing code.
  3. 03 · Repair & test I implement the patch, exercise nearby regression and failure paths, and record evidence.
  4. 04 · Handoff You receive the patch, test results, remaining risks and concise operating notes.

Scope boundaries

Clear limits protect the result.

  • One repository, workflow, environment or bounded feature per milestone unless agreed otherwise.
  • Staging, test accounts and least-privilege access are preferred.
  • New requirements, unrelated defects and extra integrations require a change order.
  • Third-party outages, account restrictions and undocumented vendor changes cannot be guaranteed away.
  • Unlimited development, 24/7 support and open-ended revisions are excluded.
  • One correction round covers a mismatch against the written acceptance criteria.

Safety and access

Access follows diagnosis—not first contact.

I do not accept credentials in an initial message. I will not bypass access controls, test without authorization, use live sensitive data unnecessarily, or make unsupported compliance promises.

I first request logs, redacted errors, reproduction steps and a dependency map. If access is needed, it should be time-limited and least-privilege. Production changes require the owner’s explicit approval and rollback plan.

Acceptance first: work starts only after deliverables, required inputs, price, timeline and acceptance criteria are written and the milestone is funded.

Start with the blocker

Send the smallest reproducible version of the problem.

Include what fails, what should happen, the exact error or reproduction steps, stack, environment, deadline and acceptance condition. Do not include passwords, API keys, private customer data or production exports.
