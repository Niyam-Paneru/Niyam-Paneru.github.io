DentSignal engineering case study · Async reliability

Two response owners made one broken turn.

Competing paths could produce duplicate output, suppress recovery without playable bytes, or leave a later request waiting indefinitely.

Area
Async state, streaming events, cancellation
Evidence
PR-reported historical evidence
Boundary
Controlled tests, not latency promise

01 / Problem

Control events were mistaken for usable output.

Multiple components could claim the same response. A control event could cancel recovery even when no playable bytes had arrived, and queued speculative work survived new input.

Root cause: ownership and cancellation depended on ambiguous activity rather than explicit output evidence.

02 / Repair

One owner. Explicit evidence. Bounded fallback.

  • Preserved conversation state while selecting one primary response owner.
  • Disabled speculative turns and cleared queued work on new caller activity.
  • Cancelled recovery only when playable output arrived.
  • Added a 2.5-second bounded fallback for the no-output path.

03 / Validation

Successive regressions followed the state machine.

The three cited PRs report 304, 306, and 310 focused or broader tests across the successive fixes, plus formatting and diff checks.

Before: duplicate, cut, or stalled responses. After: one primary owner with evidence-triggered cancellation and bounded recovery at the cited revisions.

04 / Rollback

Revert the isolated response-control sequence.

Rollback is a commit revert or prior backend image. The sequence has no database migration; response ownership and fallback behavior should be rolled back together.

05 / Sources

Exact merged change record.
