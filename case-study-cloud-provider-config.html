DentSignal engineering case study · Cloud integration

The socket opened. The provider still rejected the session.

Azure-hosted telemetry showed transport success followed by a settings rejection and zero emitted media frames. The repair stayed at the application-protocol layer.

Area
Azure telemetry, Python, WebSockets
Evidence
PR-reported historical evidence
Boundary
Application config, not Azure migration

01 / Problem

Transport health hid a protocol failure.

The WebSocket connected, but provider settings were rejected before useful output. A first threshold-only correction left the unsupported payload shape intact.

Root cause: configuration serialization included a nested object the provider did not accept.

02 / Repair

Use telemetry to reduce the hypothesis.

  • Separated connection success from post-connect settings acceptance.
  • Removed the unsupported nested object rather than tuning unrelated thresholds.
  • Constrained values to the provider-valid range.
  • Added payload-shape and override regressions.

03 / Validation

A synthetic handshake proved the narrow correction.

PR 224 reports 182 focused tests, 306 broader tests, Ruff, secret and external-service guards, plus a controlled no-audio handshake that returned SettingsApplied.

Before: rejected settings and zero frames. After: the provider accepted the controlled handshake at the cited revision.

04 / Rollback

Restore the prior backend image or revert the commits.

The repair was bounded to backend configuration and its tests. Rollback does not require a database migration or infrastructure replacement.

05 / Sources

Exact merged change record.

PR 222 / 9191216 and PR 224 / 73e4c93.

This is engineering proof, not customer proof. It does not claim an Azure infrastructure migration, a real customer interaction, or current production status. Validation is PR-reported historical evidence.
