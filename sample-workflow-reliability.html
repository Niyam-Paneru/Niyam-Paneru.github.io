Synthetic demonstration: Copper Finch Software, its systems, event IDs, timings and evidence are fictional. No customer engagement or production result is implied. The design is realistic, but a real repair would require reproduction in the buyer’s authorized environment.

Executive summary

Duplicate delivery was treated as a new business event.

The fictional billing provider retried subscription events whenever the workflow did not acknowledge within ten seconds. The original n8n flow waited for CRM and notification branches before responding, had no durable idempotency record, and retried permanent validation errors. A transient CRM timeout therefore produced both delayed acknowledgements and duplicate contacts.

The synthetic patch separates receipt from processing: validate and persist the event once, return 202 Accepted, then process durable per-destination work. A unique provider event ID prevents duplicate business effects; bounded retry with jitter handles transient faults; permanent failures enter a review queue with an alert.

Failure symptoms

What an operator observed.

  • Two CRM contacts and two onboarding messages for one subscription activation.
  • Some valid subscriptions missing from the CRM after a temporary outage.
  • Provider webhook logs showing several deliveries of the same event ID.
  • n8n executions marked failed when only the CRM branch failed.
  • No single view showing whether a delivery was pending, retried, complete or abandoned.

Reproduction

Minimal steps that expose the fault.

  1. Send a valid, signed synthetic subscription.activated event with provider event ID evt_demo_1042.
  2. Configure the CRM stub to delay for twelve seconds on the first request—longer than the provider acknowledgement window.
  3. Observe that the original flow withholds its response and the provider repeats the same event.
  4. Restore the CRM stub and allow both executions to finish.
  5. Observe two create requests and two notification requests for one provider event.

Synthetic evidence excerpt — 09:14:07 delivery 1 timed out; 09:14:19 delivery 2 accepted; CRM create calls for external_account=acct_demo_77: 2.

Root cause

Acknowledgement, orchestration and business effects were coupled.

  1. No idempotency boundary: the flow did not claim the provider event ID in durable storage before calling downstream systems.
  2. Slow acknowledgement: it returned success only after all downstream branches completed, so a healthy provider retried during a slow dependency.
  3. Non-idempotent destination action: the CRM branch used “create” rather than upsert against a stable external account key.
  4. Unsafe retry classification: timeouts, 429/5xx responses and permanent 400 validation errors shared one generic retry path.
  5. Weak observability: execution logs did not expose per-event and per-destination state or alert after exhaustion.

Patch outline

Implementation decisions.

  • Verify the webhook signature against the unmodified request body before parsing.
  • Store only the fields required for processing; record a payload hash for duplicate mismatch detection.
  • Enforce a unique database constraint on provider plus provider event ID.
  • Use destination records with their own state, attempt count and next-attempt time.
  • Use exponential backoff with jitter for timeouts, 429 and selected 5xx responses; honor Retry-After when present.
  • Do not retry authentication, permission or schema errors without an explicit rule.
  • Upsert the CRM record by stable external account ID and attach the source event ID.
  • Deduplicate alerts by destination task plus failure cycle so retries do not page repeatedly.

Remaining risks

What the patch does not magically solve.

  • A provider can reuse an event ID incorrectly; a stored payload hash detects but cannot resolve that contract violation.
  • A downstream API may accept a request and time out before responding. Destination-level idempotency keys or safe lookup-before-retry are still required.
  • Out-of-order subscription events need a version or occurred-at rule; event-ID deduplication alone does not order them.
  • Retention, redaction, queue capacity, rate limits and recovery objectives need environment-specific decisions.
  • Changes to provider schemas and authentication require monitoring and contract tests.

Handoff checklist

Operator-ready closeout.

  • Acceptance criteria and tested event types are recorded.
  • Test payloads are synthetic and contain no customer data.
  • Secret values remain in the approved secret store, not workflow exports.
  • Idempotency key, retention period and payload-hash behavior are documented.
  • Retryable and permanent status classes are listed.
  • Attempt limit, backoff schedule and alert threshold are documented.
  • Review and authorized requeue steps are demonstrated.
  • Rollback or workflow-disable procedure is recorded.
  • Dashboard queries for pending, review and oldest-event age are available.
  • Known risks and owner for each follow-up are written down.

Synthetic demonstration conclusion: the fictional acceptance tests pass, but production readiness would still depend on environment-specific load, security, data, monitoring and recovery review.
