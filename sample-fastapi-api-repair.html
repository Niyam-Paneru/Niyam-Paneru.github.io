Executive summary

Known client errors became 500s, and one failed request could partially commit.

The fictional member-invite endpoint returned 500 for malformed roles, mixed-case duplicate emails and expired tokens. A forced audit failure could occur after the membership row had already committed.

The repair validates input before business logic, preserves correct authentication and permission responses, normalizes invite identity, writes membership and audit rows in one transaction, relies on database uniqueness for concurrency safety, and maps a known conflict to 409.

Failing endpoint

POST /v1/projects/{project_id}/members

Request: { "email": "Avery@example.test", "role": "owner-admin" }

Before: 500 { "detail": "Unable to invite member" }

Expected: 422 with a field-level role validation error

The reserved .test address and all identifiers are synthetic.

Failure symptoms

Four defects shared one endpoint boundary.

  • Different email letter case bypasses the application pre-check and reaches a unique constraint.
  • An unsupported role reaches a database check rather than returning request validation.
  • An expired JWT is caught by a broad handler and converted to a generic 500.
  • Membership commits before an audit insert, so audit failure can leave partial state.
  • Concurrent duplicates produce one 201 and one unhelpful 500.
  • Unexpected errors have no stable correlation reference.

Reproduction

Minimal test sequence.

  1. Seed fictional project prj_demo_18 and its owner.
  2. Invite avery@example.test; observe 201.
  3. Invite Avery@example.test; observe unique-constraint 500.
  4. Send role owner-admin; observe database-check 500.
  5. Send a valid body with an expired token; observe generic 500.
  6. Force the audit repository to fail after membership flush; observe a partial row before repair.
  7. Send two concurrent invites for one normalized email; observe one 201 and one 500.

Validation, permission, transaction and error boundaries were blurred.

Repair approach

Make every boundary return truthful behavior.

  1. Validate email and allowed role values in the request model.
  2. Resolve authentication and invite permission before opening the write transaction.
  3. Normalize the invite identity once using the documented product rule.
  4. Add membership and audit records inside one transaction and flush before success.
  5. Keep the database unique constraint as the authoritative concurrent-duplicate boundary.
  6. Translate only the named duplicate constraint to 409; preserve 401, 403, 404 and 422.
  7. Return a safe correlation ID for unexpected 500s and keep secrets out of logs and responses.

Failure-path evidence

A failed audit write leaves no partial membership.

  • Response contains only a safe error code and correlation ID.
  • Membership and audit counts both remain zero.
  • Structured server log contains the same reference and exception class.
  • Log excludes the bearer token, database URL and complete request headers.
  • Retry after dependency recovery can succeed exactly once.

Synthetic observation: all listed assertions pass.

Remaining risks

The endpoint repair is not the whole product.

  • Email identity rules differ; normalization must match the product contract.
  • External invitation delivery needs an outbox or another idempotent post-commit design.
  • A 409 can disclose membership if permission checks are too broad.
  • Constraint names and migrations must match the deployed schema.
  • Rate limiting, invitation expiry, bulk invites, load and production release are excluded.

Handoff checklist

Contract, transaction and evidence recorded.

  • Endpoint, accepted roles and response contract are documented.
  • Authentication and authorization order is explicit.
  • Normalized identity rule and database constraint are recorded.
  • Membership and audit writes share one transaction.
  • Known conflicts map to 409; invalid bodies map to 422.
  • Unexpected errors return a safe correlation reference.
  • Duplicate, concurrent-duplicate and rollback tests pass.
  • Production release and rollback remain owner-approved.
