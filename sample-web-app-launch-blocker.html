Executive summary

Client navigation hid a broken direct-load path.

The fictional billing page worked when opened from an already-loaded dashboard but failed on a direct link, refresh and production prerender. A migrated App Router page still relied on the legacy Pages Router and a transient client store for the workspace identifier.

The repair makes the URL parameter the source of truth, validates it on the server, loads authorized workspace data explicitly, and renders deliberate loading, access, not-found and dependency-failure states. The synthetic production preview, mobile and nearby regression checks pass.

Visible symptom

Blank billing panel after refresh or shared link.

  • Dashboard-to-billing navigation usually succeeds.
  • A new tab or browser refresh requests an undefined workspace.
  • The page remains on a spinner and then throws a render error.
  • The production build fails while prerendering the migrated route.
  • The mobile error action overflows its card at 390 px.

GET /api/workspaces/undefined/billing-summary 404

TypeError: Cannot read properties of null (reading 'plan')

Error: NextRouter was not mounted — prerendering /workspaces/[workspaceId]/billing

Reproduction

Minimal steps that expose the blocker.

  1. Use a clean browser profile with no active-workspace state.
  2. Sign in as the fictional authorized workspace owner.
  3. Open /workspaces/ws_demo_17/billing directly.
  4. Observe a request containing undefined and a permanent spinner.
  5. Open the dashboard first, select the workspace, then navigate to billing; observe that it works.
  6. Run a clean production build and observe the route prerender error.

Root cause

The URL was not the route's source of truth.

  1. Wrong router contract: the App Router page expected router.query.workspaceId from the legacy Pages Router.
  2. Transient identity: request construction preferred a client store that only the dashboard populated.
  3. Unsafe render: the component dereferenced summary.plan before a successful response existed.
  4. Fixed-width fallback: the error action was wider than the mobile viewport.

The backend returned correct data for valid workspace identifiers. Changing it would not repair the route and was kept outside scope.

Code-level repair

Validate the route identity before rendering the client form.

  • Receive and validate params.workspaceId in the server page.
  • Resolve the authenticated user and load only the authorized workspace.
  • Pass explicit workspaceId and initial summary data into the client component.
  • Remove the client store from API URL construction.
  • Render explicit unauthenticated, forbidden, not-found, empty and dependency-failure states.
  • Replace the fixed-width fallback with fluid sizing and wrapping actions.

Synthetic repair shape: server params → validate → authorize → fetch → explicit client props

Remaining risks

What this bounded repair does not cover.

  • Other migrated workspace routes may still depend on transient client state.
  • The backend must continue to enforce workspace authorization.
  • Billing-data cache freshness needs an environment-specific policy.
  • Provider outages can still make billing information unavailable.
  • Payment writes, migration, load testing and production release are excluded.

Handoff notes

A closeout another engineer can inspect.

  • Repair boundary and acceptance criteria are written.
  • Direct load, navigation and refresh use one route identity.
  • Response states 200, 401, 403, 404 and 503 were exercised.
  • No secret values appear in evidence.
  • Mobile, keyboard and preview checks are recorded.
  • Production release remains owner-approved.
